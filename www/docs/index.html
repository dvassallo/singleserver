The Single Server command contract.

Single Server itself is a small Go binary. It installs a systemd daemon that listens on 127.0.0.1, keeps apps.yml as the deploy allowlist, receives GitHub push webhooks through your server's public setup URL, checks out the pushed commit with a GitHub App token, renders Kamal config from server-side conventions, and runs the deploy on the same machine. Docker containers, Cloudflare tunnel routes, app DNS records, env files, storage mounts, backups, and deploy logs are managed as server resources.

Quickstart

Start with any Linux server from Hetzner, DigitalOcean, AWS, Azure, or another provider. Run every command over SSH on that server.

SSH to your Linux server.

ssh root@203.0.113.10

Install and set up Single Server.

curl -fsSL https://singleserver.com/install.sh | sh

Add your apps.

singleserver add https://github.com/you/my-app

Check what is running.

singleserver status

After that, pushing to the configured branch should deploy automatically.

Concepts

  • Your server The server is both the deploy controller and the runtime host. The daemon, Docker, Kamal, Tailscale, Git checkouts, deploy credentials, app config, persistent storage, backups, and logs all live on this one machine. That is why normal operations happen from the server instead of across repo settings, CI runners, and separate dashboards.
  • The daemon The daemon listens locally on 127.0.0.1. Tailscale Funnel exposes the small public surface it needs: the GitHub setup page and signed GitHub webhooks. When a webhook arrives, the daemon verifies the signature, checks the repository and branch against apps.yml, fetches the exact pushed SHA, runs the deploy, writes logs, and reports the result.
  • GitHub App The GitHub App is how the server receives push events and reads code. It sends signed webhooks to your server, issues short-lived installation tokens for repository fetches, and lets the daemon write pending, success, and failure commit statuses back to GitHub. GitHub triggers the deploy, but the deploy runs on your server. doctor verifies that the GitHub App webhook URL points at this server's public setup URL.
  • apps.yml This is the central app registry and deployment allowlist. Installing the GitHub App on a repository only grants access; it does not make that repository deployable. A pushed repository must also match an app entry in /etc/singleserver/apps.yml, and the pushed branch must match that app's configured branch.
  • App containers Every app runs in its own Docker container. Single Server prepares the checkout and generated deploy config, then Kamal builds the image, starts the replacement container, waits for the readiness path, and moves traffic only after the new container is ready. Kamal's proxy binds to the server loopback interface, so app traffic comes from cloudflared instead of exposing the proxy directly on the public internet.
  • Tailscale Tailscale provides private server access and the public webhook URL. Tailscale SSH lets operators reach the server without managing SSH keys on every workstation. Tailscale Funnel forwards GitHub setup and webhook requests from the server's *.ts.net URL to the daemon's local port.
  • Cloudflare Tunnel Cloudflare is the public edge for app domains. Single Server creates proxied CNAME records to a Cloudflare Tunnel, keeps cloudflared routes in sync, and lets Cloudflare handle public TLS, proxying, CDN features, and DDoS protection while the server IP stays hidden.
  • Central state Domains, server-side environment files, storage mounts, backups, deploy logs, and provider credentials are server concerns. Repositories stay focused on application code, with build details either in a repo Dockerfile or in explicit runtime settings stored on the server. When an app needs an exception, the exception belongs in apps.yml or a server-side command, not in GitHub Actions secrets or a repo-specific runner.

Commands

Commands are grouped by setup, app operations, and server management. Each command owns its arguments, subcommands, and options directly below the command form, so there is no separate options table to cross-reference.

Setup

Install curl -fsSL https://singleserver.com/install.sh | sh

Installs Docker, Kamal, Tailscale, cloudflared, the hosted singleserver Linux binary, the singleserver.service daemon, and base config files. Then it runs first-run setup: connects or repairs Tailscale SSH and Funnel, connects Cloudflare Tunnel for app domains, prints the GitHub App setup URL once a Funnel URL exists, and runs diagnostics. In an interactive SSH session it prompts for Tailscale login and a Cloudflare API token; unattended installs can pass TAILSCALE_AUTHKEY and CLOUDFLARE_API_TOKEN. The installer is safe to rerun.

Repair Tailscale connection singleserver tailscale connect [options]

Connects or repairs Tailscale on the server, enables Tailscale SSH when possible, starts Tailscale Funnel for the daemon's local port, and stores the resulting *.ts.net URL as the GitHub setup and webhook URL.

singleserver tailscale connect options

  • --auth-key <key>Use a Tailscale auth key for unattended server joins. Prefer TS_AUTHKEY or TAILSCALE_AUTHKEY in automation so the key does not land in shell history.
  • --hostname <name>Set the Tailscale machine name during an unattended join.
Repair GitHub connection singleserver github connect

Prints the GitHub App setup URL again, creates a setup token if needed, and restarts the daemon so the browser setup page can create or reconnect the GitHub App. The GitHub App needs repository contents read, commit statuses write, and push events.

Repair Cloudflare Tunnel singleserver cloudflare connect [options]

Connects Cloudflare as the app-domain ingress provider. It creates or reuses a Cloudflare Tunnel, writes the cloudflared config and service, and prepares future app-domain commands to create proxied CNAME records and tunnel routes.

singleserver cloudflare connect options

  • --account <id>Cloudflare account to use when the API token can access multiple accounts. App domains still resolve their Cloudflare zones per hostname.
  • --tunnel <name>Cloudflare Tunnel name. Defaults to a host-specific name like singleserver-my-host.

Apps

Add an app singleserver add https://github.com/owner/repo [options]

Adds a GitHub repository, verifies app access, writes apps.yml, validates generated deploy config, configures app routing, deploys the current branch tip, and prints the live URL. In an interactive SSH session, Single Server prompts for missing runtime settings, readiness path, optional external healthcheck URL, and whether to deploy immediately. It also prints the equivalent non-interactive command. In scripts, pass the same answers as flags.

singleserver add arguments

  • <github-url>Required. A URL like https://github.com/owner/repo. Single Server stores it as owner/repo.

singleserver add generated Dockerfile examples

  • --runtime static --static-dir distServe an already-built static directory with nginx. --static-dir defaults to the repository root.
  • --runtime node --install "npm ci" --build "npm run build" --static-dir distRun explicit Node commands, then serve the named static output directory with nginx.
  • --runtime bun --install "bun install --frozen-lockfile" --build "bun run build" --start "bun run start" --app-port 3000Build and run a Bun web process. Dynamic Node/Bun apps must provide both --start and --app-port.

singleserver add interactive prompts

  • Dockerfile foundSingle Server uses the repo Dockerfile as-is, then asks for the readiness path, optional external healthcheck URL, and deploy confirmation.
  • No Dockerfile foundSingle Server asks for static, node, or bun, then asks only the follow-up questions needed for that runtime.
  • Equivalent commandAfter the prompts, Single Server prints the flag-based command that would make the same app entry without prompting.

singleserver add options

  • --branch <name>Deploy only this branch. Defaults to the repository default branch.
  • --runtime static|node|bunGenerate a temporary Dockerfile when the repository does not have one. Repos with their own Dockerfile should omit this.
  • --install <cmd>Optional install command for generated Node/Bun Dockerfiles, such as npm ci or bun install --frozen-lockfile.
  • --build <cmd>Optional build command for generated Node/Bun Dockerfiles, such as npm run build or bun run build.
  • --start <cmd>Start command for dynamic Node/Bun apps. Do not use this for static output; nginx serves static output.
  • --static-dir <dir>Static files to serve from the repo or from the output of the Node/Bun build. Defaults to . for --runtime static. Must be a relative path inside the repo.
  • --app-port <port>Container port used by the generated Kamal proxy. Defaults to 80. Required for dynamic Node/Bun generated Dockerfiles.
  • --healthcheck-path <path>Path Kamal uses to decide when the replacement container is ready. Defaults to / for normal app containers and /up for generated static output.
  • --healthcheck <url>Optional external URL Single Server checks after deploys and in status/doctor. If omitted, the external check is treated as assumed healthy.
  • --name <app>Override the app name. Defaults to the repository name lowercased.
  • --no-deployConfigure the app but wait for the next push or manual deploy.
Edit an app singleserver edit <app|owner/repo|github-url> [options]

Changes an app after it has been added. With no flags in an interactive SSH session, Single Server prompts from the current apps.yml settings and prints the equivalent non-interactive command. In scripts, pass the same settings as flags. By default, the config change deploys immediately; use --no-deploy to stage the change.

singleserver edit arguments

  • <app|owner/repo|github-url>Required. The app name, configured repo, or full GitHub URL for an app already listed in apps.yml.

singleserver edit common examples

  • singleserver edit my-appOpen the interactive editor for the app's build mode, readiness path, external healthcheck, and deploy choice.
  • singleserver edit my-app --healthcheck-path /readyChange the container readiness path Kamal checks before moving traffic.
  • singleserver edit my-app --dockerfile --no-healthcheckSwitch back to the repository Dockerfile and clear the optional external healthcheck URL.
  • singleserver edit my-app --runtime static --static-dir publicUse a generated static-site Dockerfile and serve public.

singleserver edit options

  • --branch <name>Change the production branch for the app.
  • --dockerfileUse the repository Dockerfile and clear generated Dockerfile runtime settings.
  • --runtime static|node|bunUse a generated Dockerfile. Add runtime-specific flags just like singleserver add.
  • --install <cmd>Set or replace the generated Node/Bun install command.
  • --build <cmd>Set or replace the generated Node/Bun build command.
  • --start <cmd>Set or replace the generated Node/Bun process start command.
  • --static-dir <dir>Set the static directory for generated static output.
  • --app-port <port>Set the app container port. Dynamic generated Node/Bun apps need this when the current app does not already have an explicit port.
  • --healthcheck-path <path>Set the container readiness path. Defaults stay / for app processes and /up for generated static output.
  • --healthcheck <url>Set the optional external healthcheck URL used after deploys and by status/doctor.
  • --no-healthcheckClear the optional external healthcheck URL. Container readiness still runs during deploys.
  • --no-deployWrite apps.yml without deploying immediately.
Deploy manually singleserver deploy <owner/repo|app> [ref]

Deploys a configured app immediately. If ref is omitted, it deploys the configured branch or the repository default branch. Manual deploys use the same checkout, build, rollout, optional external healthcheck, and GitHub status path as push-triggered deploys.

singleserver deploy arguments

  • <owner/repo|app>Configured repository or app name to deploy.
  • [ref]Optional branch, tag, or commit SHA. If omitted, Single Server uses the app's configured production branch.
List apps singleserver list

Prints configured apps, their repositories, branches, domains, external healthcheck setting, and current health state.

Check status singleserver status

Summarizes daemon health, app deploy status, app domains, runtime state, and optional external healthchecks. This is the quick operational view.

Run diagnostics singleserver doctor [app]

Runs deeper checks for daemon health, config validity, deploy user SSH, local registry, Tailscale, GitHub App access, generated deploy config, checkout state, recent deploy outcome, disk, Docker, Cloudflare Tunnel routes, Cloudflare DNS targets, resolver DNS, and optional external healthchecks.

View logs singleserver logs [app] [options]

Shows deploy logs by default. Filter by app name when provided, use --runtime for app container logs, --daemon for the full Single Server journal, and --follow to stream.

singleserver logs options

  • [app]Optional app name or repository. Without it, Single Server shows recent deploy logs for all apps.
  • --runtimeShow logs from the app container instead of the Single Server daemon journal. Requires an app.
  • --daemonShow the full Single Server service journal instead of only deploy lines.
  • --followStream new log lines instead of printing the recent tail.
Inspect deploy config singleserver render-deploy <owner/repo|app>

Prints the generated Kamal deploy.yml for a configured app. Use this when you want to see exactly what conventions produced before a deploy.

Remove an app singleserver remove <app> [options]

Removes an app from config, stops containers, removes routes, and removes matching DNS records. Repo checkouts and persistent storage are kept by default and deleted only with explicit confirmation.

singleserver remove options

  • --delete-storage --yesDelete the app's persistent storage directory after removing config and stopping containers.
  • --delete-repo --yesDelete the app's repository checkout after removing config and stopping containers.

Management

Manage domains singleserver domains <subcommand> ...

Adds, removes, lists, and verifies app domains. Domain changes update central config, Cloudflare DNS and tunnel routes when connected, and redeploy the app so Kamal picks up the new proxy config.

singleserver domains subcommands

  • add <app> <domain>Add the hostname to the app, create or update the proxied Cloudflare DNS record and tunnel route, and redeploy the app.
  • remove <app> <domain>Remove the hostname from the app, remove the matching Cloudflare DNS record and tunnel route, clear a matching legacy auto-healthcheck when present, and redeploy the app.
  • list [app]Print all configured domains, optionally scoped to one app.
  • verify [app]Check resolver DNS and Cloudflare DNS records when available, optionally scoped to one app.

singleserver domains add/remove options

  • --no-deployUse with add or remove to update config, Cloudflare DNS, and tunnel routes without redeploying immediately.
Manage environment singleserver env <subcommand> <app> ...

Sets, lists, and removes app environment variables and secrets on the server. Secrets stay out of GitHub repositories and are injected into the app on the next deploy.

singleserver env subcommands

  • set <app> KEY=valueWrite or replace one app environment value in the server-side env file, then print the deploy command that will apply it.
  • list <app>Print environment values currently stored for the app.
  • unset <app> KEYRemove one app environment value from the server-side env file, then print the deploy command that will apply the removal.
Enable storage singleserver storage enable <app> [options]

Creates persistent storage for SQLite databases, uploads, and other app-owned files, updates app config, and redeploys the app so Kamal mounts it into the container.

singleserver storage enable options

  • --mount <path>Container mount path. Defaults to /storage.
  • --path <path>Host storage path. Defaults to /srv/storage/<app>.
  • --no-deployStage the storage config without applying it to the running app immediately.
Back up an app singleserver backup <app>

Creates a timestamped .tar.gz backup from the app's configured persistent storage. SQLite databases are copied through SQLite's backup API before the archive is written.

Restore an app singleserver restore <app> <backup-id-or-path> --yes [options]

Restores an app backup after explicit confirmation, replaces the storage directory, keeps the previous storage directory as a rollback copy, and restarts the app containers with the restored data.

singleserver restore options

  • --yesRequired. Confirms that the current storage directory should be replaced.
  • --no-restartRestore files without stopping or restarting app containers. Use only when you have already stopped the app yourself.
Upgrade Single Server singleserver upgrade

Downloads the latest release, restarts the service, and runs singleserver doctor to verify the server after the upgrade.

Configuration

Single Server is convention over configuration. The normal path is to add an app with its GitHub URL and, when the app should be public, its domain. From that input, Single Server infers the app name, checkout path, deploy branch, readiness path, generated Kamal config, and the default storage location when storage is enabled.

The conventions are meant to be predictable rather than hidden. /etc/singleserver/apps.yml is the central override file when an app needs to differ from the defaults, such as a non-default branch, multiple domains, a different container port, a custom readiness path, an optional external healthcheck URL, or a storage mount.

Conventions

  • Repository A GitHub URL like https://github.com/owner/repo is normalized to owner/repo in apps.yml. The URL form is the user-facing command shape; the compact form is the stored config shape.
  • App name The app name defaults to the repository name, lowercased. It must be unique on the server because it becomes the stable identifier used for the checkout path, generated Kamal service name, image name, container names, logs, and storage paths.
  • Branch Deploys default to the repository default branch. Set branch only when production deploys should follow a different branch, such as main, master, or production.
  • Domain Domains belong to apps, not to the server setup. Add the hostname you want, such as app.example.com, during singleserver add or later with singleserver domains add. When Cloudflare is connected, Single Server finds the Cloudflare zone that owns that hostname, writes a proxied CNAME to the server's tunnel, keeps the tunnel route in sync, and rejects hostnames already assigned to another app.
  • Checkout Repositories are checked out under /srv/repos/<app>. Before each deploy, Single Server fetches the exact pushed SHA so production matches the GitHub event that triggered it.
  • Build If the repository has a Dockerfile, Single Server uses it. If it does not, Single Server can generate a temporary Dockerfile from explicit settings in apps.yml: runtime, optional install/build, optional static_dir, and for dynamic Node/Bun apps, start plus app_port. The generated Dockerfile is written only inside the server checkout during deploy and is removed afterward.
  • Kamal config Single Server generates Kamal config from conventions and overrides. The generated proxy config binds Kamal to 127.0.0.1 because Cloudflare Tunnel is the public app ingress. A repo-level config/deploy.yml should only be needed for unusual apps that opt out of the generated config path.
  • Port Apps listen on port 80 by default. Static generated Dockerfiles always serve nginx on port 80. Dynamic generated Node/Bun apps must set --app-port or app_port so Single Server does not guess which port the process listens on.
  • Readiness and healthchecks Kamal still needs a path to decide when the replacement container is ready before it moves traffic. For normal app containers, Single Server defaults that readiness path to /. For generated static output, it defaults to /up and the generated nginx image serves that path automatically. A public healthcheck URL is optional; when it is absent, Single Server reports the external check as assumed healthy instead of failing the app.
  • Secrets Secrets live on the server in app env files. They are injected into generated Kamal config without requiring GitHub repository secrets, deploy keys, or GitHub Actions workflows.
  • Storage Persistent app data defaults to /srv/storage/<app> and mounts into the container at /storage. This is the expected place for SQLite databases, uploads, and other app-owned data.
  • Backups Backups are written under /srv/backups/<app>. Storage-enabled apps can be backed up and restored from the server without adding app-specific backup code to the repository.
  • Logs Deploy logs and daemon logs live in the systemd journal for singleserver.service. Runtime logs come from the app containers and are surfaced through singleserver logs <app> --runtime.

Apps Config

The shortest app entry is just a repository name.

apps:
  - dvassallo/sillyface-games
  - dvassallo/fullsend

Use an object only when an app needs overrides.

apps:
  - repo: smallbets/userbase-homepage
    branch: master
    runtime: node
    install: npm ci
    build: npm run build
    static_dir: dist
    hosts:
      - userbase.com
      - www.userbase.com
    app_port: 80
  • repoRequired. GitHub repository as owner/repo. This is the only field required for a conventional app.
  • nameDefault: repository name. Lowercase app identifier used for checkout, service, image, container names, logs, and storage. App names must be unique; set this when two repositories share the same repo name.
  • branchDefault: repository default branch. Only pushes to this branch deploy automatically. Manual deploys can still name a specific ref.
  • pathDefault: /srv/repos/<app>. Checkout directory on the server.
  • hostsDefault: none. Public hostnames for generated Kamal proxy config, Cloudflare DNS, and Cloudflare Tunnel routes. Hosts must be unique across apps.
  • runtimeDefault: absent. Set to static, node, or bun only when the repository does not have a Dockerfile and Single Server should generate one during deploy.
  • installDefault: absent. Optional install command for generated Node/Bun Dockerfiles. Single Server stores and runs exactly the command you provide.
  • buildDefault: absent. Optional build command for generated Node/Bun Dockerfiles. Use this with static_dir for static sites that need a build step.
  • startDefault: absent. Start command for dynamic generated Node/Bun apps. Required when runtime is node or bun and static_dir is not set.
  • static_dirDefault: . for runtime: static. Relative directory to serve with nginx. For Node/Bun static builds, this is the output directory after the build command runs.
  • app_portDefault: 80. Container port that receives web traffic from Kamal.
  • healthcheck_pathDefault: / for normal apps, /up for generated static output. Container readiness path used by generated Kamal config.
  • healthcheckDefault: absent. Optional external healthcheck URL used by Single Server after deploys and in status checks. When absent, the external healthcheck is treated as assumed healthy.
  • storageDefault: absent until enabled. Persistent storage mapping with optional path and mount fields.

Generated Kamal Defaults

  • Service and imageDefault: app name. The generated Kamal service, image, and container identity are derived from the Single Server app name.
  • Server hostDefault: 127.0.0.1. Kamal deploys back to the same machine where Single Server is running.
  • SSHDefault: deploy user with /root/.ssh/id_ed25519. The daemon runs locally, while Kamal keeps using its normal SSH-based deploy model.
  • RegistryDefault: 127.0.0.1:5555. Images are built and pushed to the local registry so deploys do not wait on an external registry round trip.
  • BuilderDefault: local Docker builder for the server architecture. The server builds the image from the checked-out commit.
  • ProxyDefault: ssl: false behind Cloudflare Tunnel, forwarded headers on, app port 80. Kamal's proxy handles host routing to containers behind the tunnel while Cloudflare handles public TLS at the edge.
  • ReadinessDefault: / for normal apps, /up for generated static output, interval 1, timeout 1. Fast local readiness checks keep zero-downtime swaps quick for small apps.
  • TimeoutsDefault: deploy 10s, drain 1s. Defaults are tuned for fast local builds and small app containers.

App Contract

Your apps need one web port. A dedicated health endpoint is optional. They can bring their own Dockerfile, or they can let Single Server generate one from explicit runtime settings. They do not need GitHub Actions workflows, deploy keys, repo-level deploy secrets, repo-level webhooks, or a Kamal config.

Dockerfile → Used as-is when present
runtime → Generates a temporary Dockerfile when needed
port 80 → Default app port, override with --app-port
/ → Default readiness path, override with --healthcheck-path

Generated Dockerfile Examples

Static site that already has files ready to serve:

singleserver add https://github.com/you/site \
  --runtime static \
  --static-dir dist

Node site that builds static files:

singleserver add https://github.com/you/site \
  --runtime node \
  --install "npm ci" \
  --build "npm run build" \
  --static-dir dist

Bun app that runs as a web process:

singleserver add https://github.com/you/app \
  --runtime bun \
  --install "bun install --frozen-lockfile" \
  --build "bun run build" \
  --start "bun run start" \
  --app-port 3000

Deploys

Whenever you push to GitHub, your server fetches the commit, builds it locally, and uses Kamal to roll it out with zero downtime.

git push → GitHub sends a webhook to your server
allowlist → apps.yml decides whether this repo can deploy
checkout → Your server fetches the exact pushed SHA
build → Kamal builds locally with generated or repo-provided deploy config
live → Kamal rolls it out and Single Server checks health
status → GitHub gets pending, success, or failure commit status

Files On Your Server

  • /etc/singleserver/apps.ymlApp allowlist and central app config.
  • /etc/singleserver/singleserver.envDaemon environment, including local port, Tailscale Funnel public URL, setup token, and provider setup values.
  • /etc/singleserver/github-app.jsonGitHub App metadata, including app ID and webhook secret.
  • /etc/singleserver/github-app.private-key.pemGitHub App private key used to request short-lived installation tokens.
  • /etc/singleserver/tailscale.jsonTailscale hostname and Funnel URL detected by singleserver tailscale connect.
  • /etc/singleserver/cloudflare.jsonCloudflare account, tunnel, and DNS state created by singleserver cloudflare connect.
  • /srv/repos/<app>Git checkout for each app.
  • /srv/storage/<app>Persistent storage for SQLite databases, uploads, and app-owned files.
  • /srv/backups/<app>Backups created by singleserver backup.
  • journalctl -u singleserver.serviceDaemon and deploy logs.

Build Order

  • Blank server setupinstall.sh should make a new Linux server ready in one guided flow.
  • GitHub, Tailscale, and Cloudflare connectionThe installer should wrap provider setup. github connect, tailscale connect, and cloudflare connect should exist as repair commands, with Tailscale Funnel for GitHub setup/webhooks and Cloudflare Tunnel for app domains.
  • App onboardingsingleserver add https://github.com/owner/repo should infer sensible defaults, configure routing, deploy immediately, and show the live URL.
  • Domainssingleserver domains should add, remove, list, and verify app domains through Cloudflare DNS and tunnel routes.
  • Observabilitystatus, doctor, and logs should make deploy state, runtime logs, DNS, proxy, disk, Docker, and healthchecks obvious.
  • Secrets and storageenv, storage, backup, and restore should make SQLite-backed apps practical on one server.
  • Lifecycleremove and upgrade should be first-class commands with clear confirmation and rollback paths.
