The Single Server command contract.

Single Server itself is a small Go binary: no Node runtime, no database, and no per-repo runner. It uses Docker, Kamal, and Cloudflare Tunnel on your Linux server to deploy your apps.

These docs are written as the product contract. Commands marked Built work in the current codebase. Partial commands exist but do not yet match the ideal UX. Planned commands describe what still needs to be built.

Quickstart

Start with any Linux server from Hetzner, DigitalOcean, AWS, Azure, or another provider. Run every command over SSH on that server.

ssh root@203.0.113.10
curl -fsSL https://singleserver.com/install.sh | sh
singleserver init
singleserver add https://github.com/you/my-app
singleserver status

After that, pushing to the configured branch should deploy automatically.

Concepts

Your server

The control plane. Single Server, Docker, Kamal, cloudflared, checkouts, config, secrets, and deploy logs live here.

GitHub App

The event source and deploy credential provider. Push events wake your server; installation tokens fetch code and set commit status.

apps.yml

The allowlist. Only repositories in this file can deploy, even if the GitHub App is installed more broadly.

App containers

Each project runs in its own Docker container. Kamal builds, starts, healthchecks, and swaps traffic.

Cloudflare Tunnel

The expected public ingress path. App domains route through Cloudflare to your server without direct TLS management on the host.

Central secrets

Deploy secrets belong on the server, not in every GitHub repo. Repos should only need app code and a Dockerfile.

Commands

Setup

Planned Install curl -fsSL https://singleserver.com/install.sh | sh

Installs Docker, Kamal, cloudflared, the singleserver CLI, the singleserver.service daemon, and base config files. It should be safe to rerun.

Planned singleserver init singleserver init

Initializes your server. It should connect Cloudflare Tunnel and GitHub, create required secrets under /etc/singleserver, start the daemon, and finish with singleserver doctor.

Planned singleserver github connect singleserver github connect

Repair command for GitHub App setup. It should open or print a GitHub URL, request contents read and commit statuses write, and keep apps.yml as the deployment allowlist.

Planned singleserver cloudflare connect singleserver cloudflare connect

Repair command for Cloudflare. It should create or reuse a tunnel, save credentials, and prepare DNS/routing for app domains.

Apps

Partial singleserver add singleserver add <github-url> [options]

Adds a GitHub repository, verifies access, checks for a Dockerfile, writes apps.yml, validates the generated Kamal config, and should deploy the current branch tip by default.

Built singleserver deploy singleserver deploy <owner/repo> [ref]

Deploys a configured app immediately. If ref is omitted, it deploys the configured branch or the repository default branch.

Built singleserver list singleserver list

Prints configured apps, their repository, branch, and healthcheck URL.

Partial singleserver status singleserver status

Checks the local daemon and configured app healthchecks. Target behavior should summarize deploy status, domains, and runtime health.

Partial singleserver doctor singleserver doctor

Runs deeper checks. Today it checks daemon health, config validity, GitHub App access, generated deploy config, checkout state, recent deploy outcome, and healthchecks.

Built singleserver logs singleserver logs [app]

Shows recent singleserver.service journal logs, optionally filtered by app name. Target flags include --follow and --runtime.

Built singleserver render-deploy singleserver render-deploy <owner/repo>

Prints the generated Kamal deploy.yml for a configured app. Useful for debugging conventions.

Planned singleserver remove singleserver remove <app>

Removes an app from config, stops containers, removes routing, and asks before deleting persistent storage.

Add Options

Option Status What it does
<github-url> Built Accepts a URL like https://github.com/owner/repo and normalizes it to owner/repo in config.
--branch <name> Built Deploys only this branch. Defaults to the repository default branch.
--app-port <port> Built Container port used by the generated Kamal proxy. Defaults to 80.
--healthcheck-path <path> Built Path Kamal uses for container healthchecks. Defaults to /up.
--healthcheck <url> Built External URL Single Server checks after deploys and in status/doctor.
--name <app> Built Overrides the app name. Defaults to the repository name lowercased.
--host <domain> Built, advanced Current explicit host override for generated Kamal proxy config. Can be repeated. Target UX should prefer domains add or domain inference.
--path <dir> Built, advanced Overrides the checkout path. Defaults to /srv/repos/<app>.
--deploy Built, temporary Current implementation deploys immediately only when this flag is passed.
--no-deploy Planned Target escape hatch once add deploys by default.
--dry-run Built Validates without writing apps.yml.

Management

Planned singleserver domains singleserver domains add <app> <domain>

Adds, removes, and lists app domains. Should update central config, Cloudflare DNS, and tunnel routing.

Planned singleserver env singleserver env set <app> KEY=value

Manages app environment variables and secrets on the server.

Planned singleserver storage singleserver storage enable <app> --mount /storage

Enables persistent storage for SQLite databases, uploads, and other app-owned files.

Planned singleserver backup singleserver backup <app>

Creates an app backup. SQLite apps should have a first-class, low-drama backup path.

Planned singleserver restore singleserver restore <app> <backup-id>

Restores an app backup after confirmation.

Planned singleserver upgrade singleserver upgrade

Downloads the latest release, restarts the service, and runs singleserver doctor.

Configuration

The main config file is /etc/singleserver/apps.yml. The shortest app entry is just a repository name.

apps:
  - dvassallo/sillyface-games
  - dvassallo/fullsend

Use an object only when an app needs overrides.

apps:
  - repo: smallbets/userbase-homepage
    branch: master
    hosts:
      - userbase.com
      - www.userbase.com
    app_port: 80
    healthcheck_path: /up
    healthcheck: https://userbase.com/up
Field Default Meaning
repo required GitHub repository as owner/repo.
name repository name Lowercase app name used for checkout, service, image, and logs.
branch repo default branch Only pushes to this branch deploy.
path /srv/repos/<app> Checkout directory on the server.
hosts empty Public hostnames for generated Kamal proxy config.
app_port 80 Port your container listens on.
healthcheck_path /up Container healthcheck path used by generated Kamal config.
healthcheck empty External healthcheck URL used by Single Server after deploys and in status checks.

Server Environment

Variable Default Purpose
SINGLESERVER_CONFIG /etc/singleserver/apps.yml Main app config path.
SINGLESERVER_STATE_DIR /etc/singleserver Directory for GitHub App credentials and Single Server state.
SINGLESERVER_PORT 8787 Local daemon port. The daemon listens on 127.0.0.1.
SINGLESERVER_PUBLIC_URL init-managed Public webhook/setup URL. The target UX should hide this behind singleserver init.

App Contract

Your apps need a Dockerfile, one web port, and a health endpoint. They do not need GitHub Actions workflows, deploy keys, repo-level deploy secrets, repo-level webhooks, or a Kamal config.

Dockerfile → Your repo defines how it builds
port 80 → Default app port, override with --app-port
/up → Default healthcheck path, override with --healthcheck-path

Dockerfile Examples

Static site:

FROM nginx:alpine
COPY dist /usr/share/nginx/html

Node or Bun app:

FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .
ENV PORT=3000
EXPOSE 3000
CMD ["bun", "start"]
singleserver add https://github.com/you/my-node-app --app-port 3000

Deploys

Whenever you push to GitHub, your server fetches the commit, builds it locally, and uses Kamal to roll it out with zero downtime.

git push → GitHub sends a webhook to your server
allowlist → apps.yml decides whether this repo can deploy
checkout → Your server fetches the exact pushed SHA
build → Kamal builds locally with generated or repo-provided deploy config
live → Kamal rolls it out and Single Server checks health
status → GitHub gets pending, success, or failure commit status

Files On Your Server

Path Purpose
/etc/singleserver/apps.yml App allowlist and config.
/etc/singleserver/singleserver.env Daemon environment.
/etc/singleserver/github-app.json GitHub App metadata.
/etc/singleserver/github-app.private-key.pem GitHub App private key.
/srv/repos/<app> Git checkout for each app.
/srv/storage/<app> Planned persistent storage for app data.
journalctl -u singleserver.service Daemon and deploy logs.

Roadmap Status

Experience Status Notes
Central apps.yml allowlist Built Pushes for unlisted repos are ignored.
GitHub App push deploys Built Webhooks and installation tokens drive deploys.
Generated Kamal config Built Repos do not need config/deploy.yml unless they want custom behavior.
singleserver add Partial Current command supports URL input, validation, hosts, dry run, and optional deploy. Target is deploy-by-default plus domain inference.
singleserver doctor Partial Needs broader disk, Docker, proxy, DNS, and runtime checks.
Installer and init Planned Should make a blank server ready in one guided flow.
Cloudflare DNS and tunnel management Planned Cloudflare Tunnel is the assumed ingress path.
Domains, env, storage, backups, removal, upgrade Planned These should become first-class server-side commands.
